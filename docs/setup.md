## Установка

### 1. Добавьте gem в Gemfile

```ruby
# Gemfile
gem "onboard_on_rails"
```

```bash
bundle install
```

### 2. Подключите engine в маршрутах

```ruby
# config/routes.rb
Rails.application.routes.draw do
  mount OnboardOnRails::Engine => "/onboard"
  # остальные маршруты вашего приложения
end
```

Engine будет доступен по префиксу `/onboard`. Вы можете использовать любой другой путь, например `/tours` или `/onboarding`.

### 3. Запустите миграции

```bash
bin/rails onboard_on_rails:install:migrations
bin/rails db:migrate
```

Будут созданы 4 таблицы:

| Таблица | Назначение |
|---|---|
| `onboard_on_rails_tours` | Туры: настройки, таргетинг, A/B тесты |
| `onboard_on_rails_steps` | Шаги туров: селектор, текст, стили |
| `onboard_on_rails_completions` | Прогресс пользователей по турам |
| `onboard_on_rails_events` | Пользовательские события для триггеров |

> **Важно**: требуется PostgreSQL — таблицы используют `jsonb` колонки для `url_pattern`, `segment_rules`, `style_overrides` и `payload`.

### 4. Создайте инициализатор

```ruby
# config/initializers/onboard_on_rails.rb
OnboardOnRails.configure do |config|
  # Название вашей модели пользователя (строка)
  config.user_class = "User"

  # Метод на вашем ApplicationController, возвращающий текущего пользователя
  config.current_user_method = :current_user

  # Лямбда для проверки доступа к админ-панели.
  # Получает экземпляр контроллера, должна вернуть true/false.
  config.admin_auth = ->(controller) {
    controller.current_user&.admin?
  }

  # Лямбда, возвращающая хеш атрибутов пользователя для сегментации.
  # Эти атрибуты используются в правилах таргетинга туров.
  config.user_attributes = ->(user) {
    {
      plan: user.plan,
      role: user.role,
      company_size: user.company&.size,
      created_at: user.created_at.iso8601
    }
  }
end
```

#### Описание параметров конфигурации

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| `user_class` | String | `"User"` | Название модели ActiveRecord, представляющей пользователя |
| `current_user_method` | Symbol | `:current_user` | Метод на `ApplicationController`, возвращающий текущего пользователя |
| `admin_auth` | Lambda | `->(_) { true }` | Проверка доступа к админке. Получает контроллер, возвращает boolean |
| `user_attributes` | Lambda | `->(_) { {} }` | Возвращает хеш атрибутов для правил сегментации |

### 5. Добавьте мета-теги и ассеты в layout

```erb
<%# app/views/layouts/application.html.erb %>
<!DOCTYPE html>
<html>
<head>
  <title>Ваше приложение</title>

  <%# Мета-теги OnboardOnRails: user-id, mount-path, csrf-token %>
  <%= onboard_on_rails_meta_tags %>

  <%# Клиентский JavaScript и стили для показа туров %>
  <%= javascript_include_tag "onboard_on_rails/client" %>
  <%= stylesheet_link_tag "onboard_on_rails/client" %>
</head>
<body>
  <%= yield %>
</body>
</html>
```

Хелпер `onboard_on_rails_meta_tags` генерирует три мета-тега:

```html
<meta name="onboard-on-rails-user-id" content="123">
<meta name="onboard-on-rails-mount-path" content="/onboard">
<meta name="csrf-token" content="...">
```

Клиентский JS считывает их для определения пользователя, пути API и CSRF-токена. Если пользователь не авторизован (`current_user` вернул `nil`), мета-теги не выводятся и туры не загружаются.

## Аутентификация

### Как engine получает current_user

OnboardOnRails не имеет собственной системы аутентификации. Вместо этого он делегирует получение текущего пользователя вашему приложению:

1. Создаёт экземпляр `::ApplicationController` вашего приложения
2. Копирует в него текущий `request`
3. Вызывает метод, указанный в `config.current_user_method`

```ruby
# Внутренняя реализация (app/controllers/onboard_on_rails/admin/base_controller.rb)
def current_user
  host_controller = ::ApplicationController.new
  host_controller.request = request
  host_controller.send(OnboardOnRails.configuration.current_user_method)
end
```

Это работает с Devise и другими библиотеками, которые хранят сессию в cookies, потому что объект `request` содержит cookies пользователя.

### Доступ к админке (admin_auth)

Лямбда `admin_auth` вызывается перед каждым действием админ-контроллера. Если она возвращает `false`, пользователь получает HTTP 403 Forbidden.

**Пример с Devise:**

```ruby
config.admin_auth = ->(controller) {
  user = controller.current_user
  user.present? && user.admin?
}
```

**Пример с ролевой моделью:**

```ruby
config.admin_auth = ->(controller) {
  user = controller.current_user
  user.present? && user.role.in?(%w[admin superadmin])
}
```

**Пример с IP-ограничением:**

```ruby
config.admin_auth = ->(controller) {
  controller.current_user&.admin? &&
    controller.request.remote_ip.start_with?("10.0.")
}
```

### Аутентификация API

API-контроллеры (`/onboard/api/*`) используют ту же сессионную cookie, что и ваше приложение. Клиентский JavaScript отправляет запросы к API автоматически — дополнительная настройка не нужна.

Если пользователь не авторизован, API возвращает HTTP 401 Unauthorized.

CSRF-токен передаётся через мета-тег `csrf-token` и включается в POST-запросы клиентским JS автоматически.

## Создание туров через админку

### Доступ к админке

Откройте в браузере `/onboard/admin` (или ваш путь монтирования + `/admin`). Вы увидите список туров с возможностью фильтрации по статусу.

### Создание тура

Нажмите **«Новый тур»** и заполните форму:

| Поле | Описание |
|---|---|
| **Название** | Внутреннее название тура (видно только в админке) |
| **Описание** | Необязательное описание для коллег |
| **Статус** | `Черновик` — не показывается; `Активный` — виден пользователям; `В архиве` — скрыт, данные сохранены |
| **Тип триггера** | `Автоматический` — при загрузке страницы; `По событию` — при наступлении события; `Ручной` — через API |
| **Событие триггера** | Название события (только для типа «По событию»), например `first_project_created` |
| **URL паттерн** | На каких страницах показывать. Через запятую. Примеры: `/dashboard`, `/projects/*`, `/settings/**` |
| **Тема** | Tooltip, Modal, Banner, Slideout |
| **Частота** | `Один раз` — один раз навсегда; `Каждая сессия` — раз за сессию; `Всегда` — при каждом посещении |
| **Приоритет** | Число >= 0. При нескольких подходящих турах показывается тур с наибольшим приоритетом |
| **Расписание** | Необязательно. Начало и конец периода показа тура |
| **A/B тест** | Необязательно. ID эксперимента и группа (A, B, control) |

### Добавление шагов

После создания тура вы попадёте на страницу редактирования. Нажмите **«Добавить шаг»**.

| Поле | Описание |
|---|---|
| **Заголовок** | Текст заголовка подсказки (отображается жирным) |
| **Текст** | Содержание подсказки. Поддерживается HTML: `<b>`, `<br>`, `<a>`, `<code>` |
| **CSS селектор** | Элемент, к которому привяжется подсказка. Примеры: `#header`, `.sidebar-nav`, `button.btn-primary` |
| **Расположение** | Где появится подсказка: `Сверху`, `Снизу`, `Слева`, `Справа`, `По центру` (модальное) |
| **Позиция** | Порядковый номер шага в туре |
| **Тип действия** | `Следующий` — переход к следующему шагу; `Перенаправление` — переход по URL; `Пользовательское событие` — отправка события |
| **Значение действия** | URL (для перенаправления) или имя события |
| **Ждать селектор** | Необязательно. Подождать появления элемента перед показом шага |
| **URL паттерн** | Необязательно. Показывать шаг только на определённой странице |
| **Переопределение стилей** | Цвет фона, текста, кнопок, шрифт, скругление |

### Визуальный выбор селектора (Selector Picker)

На странице редактирования шага рядом с полем «CSS селектор» есть кнопка **«Выбрать»**. При нажатии:

1. Открывается iframe с целевой страницей вашего приложения
2. При наведении на элементы они подсвечиваются
3. При клике генерируется уникальный CSS-селектор
4. Селектор автоматически вставляется в поле формы

URL целевой страницы определяется из `url_pattern` тура (первый паттерн, без подстановок).

### Активация тура

Измените статус тура на **«Активный»** и сохраните. Тур начнёт показываться пользователям на целевых страницах.

## Тур на странице пользователя

### Как работает клиентский JS

1. При загрузке страницы `OnboardOnRails.TourManager.init()` запускается автоматически
2. JS считывает мета-теги: `user-id`, `mount-path`, `csrf-token`
3. Если пользователь авторизован, отправляется запрос `GET /onboard/api/tours?url=/текущий-путь&session_id=...`
4. Сервер находит подходящий тур (с наивысшим приоритетом) через `TourMatcher`
5. Если тур найден, начинается его показ

### Как подбирается тур (TourMatcher)

`TourMatcher` последовательно применяет фильтры:

1. **Статус**: только `active` туры
2. **Расписание**: текущее время между `schedule_start` и `schedule_end` (если заданы)
3. **Наличие шагов**: туры без шагов исключаются
4. **URL**: текущий путь совпадает с одним из паттернов
5. **Сегменты**: атрибуты пользователя удовлетворяют правилам
6. **Частота**: тур не был завершён/пропущен (для `once`) или не был в этой сессии (для `every_session`)
7. **Событие-триггер**: для `event`-туров — событие должно существовать
8. **A/B тест**: пользователь должен быть в нужной группе
9. **Приоритет**: из оставшихся выбирается тур с наибольшим `priority`

### Поведение проигрывателя

- **Оверлей** — затемняет страницу, вырезает «окно» вокруг целевого элемента
- **Подсказка** — позиционируется относительно элемента (top/bottom/left/right) или по центру экрана
- **Навигация** — кнопки «Назад», «Далее», «Пропустить»
- **Точки/счётчик** — до 5 шагов отображаются точки, более 5 — числовой счётчик
- **Адаптивность** — при изменении размера окна или скролле позиция пересчитывается

### Отслеживание прогресса

- При переходе к следующему шагу отправляется `POST /api/completions` со статусом `in_progress`
- При пропуске тура — статус `dismissed`
- При завершении последнего шага — статус `completed`
- Каждый запрос включает `session_id` для контроля частоты «каждая сессия»

## Темы оформления

OnboardOnRails поддерживает 4 темы. Тема задаётся на уровне тура.

### Tooltip (Тултип)

Компактная всплывающая подсказка рядом с целевым элементом. Идеально для пошаговых инструкций.

- Позиционируется относительно элемента (top, bottom, left, right)
- Небольшой размер, не перекрывает контент

### Modal (Модальное окно)

Появляется по центру экрана с затемнённым фоном. Подходит для приветствий, важных объявлений, длинного текста.

- Всегда placement = center
- Оверлей затемняет всю страницу

### Banner (Баннер)

Полоса внизу экрана на всю ширину. Для ненавязчивых уведомлений и минимальных подсказок.

### Slideout (Выдвижная панель)

Панель справа на всю высоту экрана. Для подробных инструкций с большим объёмом текста.

### Переопределение стилей

Стили можно настроить на двух уровнях:

1. **На уровне тура** (`tour.style_overrides`) — применяется ко всем шагам
2. **На уровне шага** (`step.style_overrides`) — переопределяет стили тура для конкретного шага

Доступные свойства:

| Свойство | Описание | Пример |
|---|---|---|
| `background_color` | Цвет фона подсказки | `#ffffff` |
| `text_color` | Цвет текста | `#333333` |
| `font_family` | Шрифт | `Inter, sans-serif` |
| `font_size` | Размер шрифта | `14px` |
| `border_radius` | Скругление углов | `8px` |
| `button_color` | Цвет кнопок | `#4f46e5` |

## Таргетинг и правила

### URL паттерны

Поле `url_pattern` поддерживает три формата:

| Формат | Пример | Совпадение |
|---|---|---|
| Точное совпадение | `/dashboard` | Только `/dashboard` |
| Одноуровневый glob | `/projects/*` | `/projects/1`, `/projects/new`, но не `/projects/1/edit` |
| Многоуровневый glob | `/settings/**` | `/settings`, `/settings/profile`, `/settings/team/members` |
| Регулярное выражение | `/users/\d+/edit` | `/users/42/edit` (обратный слеш включает режим regex) |

Можно указать несколько паттернов через запятую в форме или как массив в jsonb.

### Сегменты пользователей

Правила сегментации задаются в поле `segment_rules` (jsonb). Формат:

```json
{
  "logic": "and",
  "conditions": [
    { "attribute": "plan", "operator": "eq", "value": "pro" },
    { "attribute": "role", "operator": "in", "value": ["admin", "manager"] }
  ]
}
```

Доступные операторы:

| Оператор | Описание |
|---|---|
| `eq` | Равно |
| `not_eq` | Не равно |
| `in` | Входит в список |
| `not_in` | Не входит в список |
| `gt` | Больше |
| `lt` | Меньше |
| `gte` | Больше или равно |
| `lte` | Меньше или равно |

Логика `"and"` — все условия должны выполняться. `"or"` — хотя бы одно.

Атрибуты пользователя берутся из лямбды `config.user_attributes`.

### Триггеры по событиям

Для типа триггера `event` тур показывается только если у пользователя есть соответствующее событие в базе.

**Серверная отправка:**

```ruby
# Например, в контроллере или сервисе
OnboardOnRails.track_event(current_user, "first_project_created", { project_id: project.id })
```

**Клиентская отправка:**

```javascript
OnboardOnRails.trackEvent("feature_discovered", { feature: "reports" });
```

### Частота показа

| Значение | Поведение |
|---|---|
| `once` | Показать один раз. После завершения или пропуска — больше не показывать |
| `every_session` | Показать один раз за сессию браузера. Сессия определяется по `session_id` из sessionStorage |
| `always` | Показывать при каждом посещении целевой страницы |

### A/B тестирование

Для A/B теста:

1. Создайте несколько туров с **одинаковым** `ab_test_id` и **разными** `ab_test_group`
2. Например: тур A с `ab_test_id: "welcome_v2"`, `ab_test_group: "A"` и тур B с теми же `ab_test_id`, `ab_test_group: "B"`
3. Пользователи распределяются автоматически через SHA256-хеш от `user_id + ab_test_id`
4. Распределение детерминировано: один и тот же пользователь всегда попадает в одну группу

Результаты смотрите на странице статистики тура — там есть раздел «Результаты A/B теста».

### Расписание

Необязательные поля `schedule_start` и `schedule_end`:

- Если `schedule_start` задан — тур не покажется раньше этой даты
- Если `schedule_end` задан — тур автоматически перестанет показываться после этой даты
- Если оба пустые — тур активен всегда (пока статус `active`)

Полезно для анонсов функций, привязанных к дате релиза.

## Обучающие уроки (Self-Tour)

### Что это такое

OnboardOnRails содержит 3 встроенных тура, которые обучают работе с админ-панелью:

| Урок | Тема | Страница |
|---|---|---|
| Урок 1: Знакомство с панелью | Обзор навигации и основных элементов | `/onboard/admin` |
| Урок 2: Создание тура | Все поля формы создания тура | `/onboard/admin/tours/new` |
| Урок 3: Настройка шагов | Добавление шагов, селекторы, стили | Страница нового/редактирования шага |

### Как создать уроки

**Через интерфейс:**

1. Откройте `/onboard/admin/lessons`
2. Нажмите **«Создать уроки»**

**Через код:**

```ruby
OnboardOnRails::SelfTourSeeder.seed!
```

### Как работает SelfTourSeeder

`SelfTourSeeder` использует `find_or_initialize_by` по названию тура — это значит:

- При первом запуске создаёт 3 тура со всеми шагами
- При повторном запуске обновляет существующие туры (пересоздаёт шаги)
- Все уроки помечены `ab_test_id: "self_tour"` — по этому маркеру контроллер уроков находит их

### Настройка уроков

Уроки — обычные туры. После создания вы можете:

- Изменить текст шагов через админку
- Добавить или удалить шаги
- Изменить селекторы (если вы кастомизировали CSS классы админки)

> **Внимание**: повторный вызов `SelfTourSeeder.seed!` пересоздаст шаги и перезапишет ваши изменения.

### Повторное прохождение

На странице уроков (`/onboard/admin/lessons`) рядом с каждым уроком есть кнопка **«Повторить»**. При нажатии:

1. Удаляются записи о прохождении (completions) для текущего пользователя
2. Происходит перенаправление на целевую страницу урока
3. Тур запускается заново

## Статистика

### Страница статистики

Для каждого тура доступна страница статистики. Откройте её:

- Из списка туров — кнопка **«Статистика»** в строке тура
- Или по URL: `/onboard/admin/tours/:id/stats`

### Показатели

**Сводка:**

| Метрика | Описание |
|---|---|
| Начато | Сколько пользователей начали тур |
| Завершено | Сколько прошли все шаги |
| Пропущено | Сколько нажали «Пропустить» |
| Процент завершения | Завершено / Начато * 100% |

**Отказы по шагам** — таблица с количеством пользователей, пропустивших тур на каждом шаге. Помогает найти «проблемный» шаг.

**История прохождений** — последние 50 записей: пользователь, статус, последний шаг, даты начала и завершения.

**Результаты A/B теста** — если у тура задан `ab_test_id`: сравнение групп по количеству и проценту завершений.

### REST Endpoints

Все пути относительны к точке монтирования engine (по умолчанию `/onboard`).

#### GET /api/tours

Получение подходящего тура для текущей страницы.

**Параметры:**

| Параметр | Тип | Описание |
|---|---|---|
| `url` | string | Текущий путь страницы (обязательный) |
| `session_id` | string | ID сессии для контроля частоты (необязательный) |

**Ответ (тур найден):**

```json
{
  "tour": {
    "id": 1,
    "name": "Welcome Tour",
    "theme": "tooltip",
    "style_overrides": {},
    "steps": [
      {
        "id": 1,
        "position": 1,
        "title": "Welcome!",
        "body": "Let's get started...",
        "selector": "#header",
        "placement": "bottom",
        "url_pattern": null,
        "style_overrides": {},
        "action_type": "next",
        "action_value": null,
        "wait_for_selector": null
      }
    ]
  }
}
```

**Ответ (тур не найден):**

```json
{
  "tour": null
}
```

#### POST /api/completions

Создание или обновление записи о прохождении.

**Тело запроса:**

```json
{
  "tour_id": 1,
  "step_id": 3,
  "status": "in_progress",
  "session_id": "abc123"
}
```

Допустимые значения `status`: `in_progress`, `completed`, `dismissed`.

**Ответ:**

```json
{
  "completion": {
    "id": 1,
    "tour_id": 1,
    "status": "in_progress"
  }
}
```

#### POST /api/events

Отправка пользовательского события.

**Тело запроса:**

```json
{
  "name": "first_project_created",
  "payload": { "project_id": 42 }
}
```

**Ответ:**

```json
{
  "event": {
    "id": 1,
    "name": "first_project_created"
  }
}
```

### Серверный API (Ruby)

```ruby
# Отправка события
OnboardOnRails.track_event(user, "subscription_activated", { plan: "pro" })
```

Это создаёт запись `OnboardOnRails::Event` в базе. Тур с типом триггера `event` и `trigger_event: "subscription_activated"` начнёт показываться этому пользователю.

### Клиентский API (JavaScript)

```javascript
// Отправка события
OnboardOnRails.trackEvent("feature_discovered", { feature: "reports" });

// trackEvent возвращает Promise<boolean>
const success = await OnboardOnRails.trackEvent("onboarding_complete");
```

## Локализация

### Поддерживаемые языки

- Английский (`en`) — `config/locales/en.yml`
- Русский (`ru`) — `config/locales/ru.yml`

Язык определяется настройкой `I18n.locale` в вашем приложении. Все строки админ-панели используют хелпер `t()`.

### Расположение файлов локализации

```
config/locales/
  en.yml    # Английский
  ru.yml    # Русский
```

Файлы загружаются автоматически через инициализатор engine.

### Добавление нового языка

1. Скопируйте `config/locales/en.yml` в `config/locales/<locale>.yml`
2. Переведите все строки внутри ключа `<locale>:`
3. Файл будет подхвачен автоматически при следующем запуске сервера

**Пример для немецкого:**

```yaml
de:
  onboard_on_rails:
    admin:
      nav:
        brand: "OnboardOnRails"
        tours: "Touren"
        lessons: "Lektionen"
      tours:
        index:
          title: "Touren"
          new_tour: "Neue Tour"
          # ... остальные ключи
```

### Что переводится

- Навигация и заголовки админ-панели
- Метки и подсказки форм
- Статусы, типы триггеров, частоты, темы, расположения
- Flash-сообщения (уведомления при создании/обновлении/удалении)
- Ошибки валидации ActiveRecord
- Страница статистики
- Страница уроков

> **Примечание**: содержание туров (заголовки и тексты шагов) не локализуется через I18n — они вводятся вручную в админке. Для мультиязычных туров создайте отдельные туры с сегментацией по атрибуту `locale`.
